# Troubleshooting

Common issues and their solutions, organized by platform.

---

## All Platforms

### The LLM is not following the wiki schema (no frontmatter, wrong folder)

**Cause:** The system prompt is not loaded, has been truncated, or the model
is too small to follow complex structured instructions.

**Fix:**
1. Confirm the system prompt is loaded — ask: "What commands do you support?"
   The answer should name Lookup, Ingest, Research, etc.
2. If it responds generically, reload the system prompt (see your platform
   setup guide).
3. For small local models: use the compact system prompt generated by
   `WIZARD_SHORT.md`. Shorter instructions are followed more reliably.

---

### CLAUDE.md not being read (Claude Code)

**Cause:** `CLAUDE.md` is not at the vault root, or you opened a parent
folder instead of the vault folder.

**Fix:**
- Confirm `CLAUDE.md` is at the top level of the folder you opened in
  Claude Code — not in a subfolder.
- Run `ls` in your Claude Code session to verify the working directory
  is the vault root.
- Claude Code reads `CLAUDE.md` on startup. If you moved the file, restart
  the session.

```bash
# Correct structure:
~/my-second-brain/
  CLAUDE.md          ← must be here
  wiki/
  raw/
```

### Agents not routing correctly (Claude Code)

**Cause:** The routing table in `CLAUDE.md` has a typo, missing agent file,
or overlapping trigger phrases between agents.

**Fix:**
1. Open `CLAUDE.md` and check the routing table. Verify that each agent's
   file path is correct and the file exists in `.claude/agents/`.
2. Check for overlapping trigger phrases — if two agents both claim "history",
   the router may pick arbitrarily. Make trigger phrases specific.
3. Test routing with a clearly domain-specific query:
   ```
   Research the Peloponnesian War
   ```
   The orchestrator should name the agent it is routing to. If it doesn't
   name an agent, the routing table is not being read.
4. After editing `CLAUDE.md`, restart the Claude Code session.

### Wiki not updating — pages not being written to disk

**Cause:** The LLM platform does not have file-write access, or a path
error is causing the write to fail silently.

**Fix:**
- **Claude Code:** has file-write access by default. If writes are failing,
  check that you have write permissions on the vault folder. Run:
  ```bash
  ls -la ~/my-second-brain/wiki/
  ```
- **All other platforms:** these do not have file-write access. You must
  copy output manually into Obsidian. See the copy workflow in
  [03-setup-claude-desktop.md](03-setup-claude-desktop.md).
- If Claude Code reports a path error, confirm the vault root is the current
  working directory (`pwd` in a terminal).

### wiki/log.md sync conflicts (iCloud / Obsidian Sync)

**Cause:** `wiki/log.md` is appended to frequently. When iCloud or Obsidian
Sync is active, concurrent writes (LLM writing, sync uploading) create
conflict copies.

**Fix:**
- **Option A — Split to daily logs:** Instead of one `log.md`, create one
  file per day in a `wiki/logs/` folder:
  `wiki/logs/2024-03-15.md`. Instruct the orchestrator:
  > "Append log entries to `wiki/logs/YYYY-MM-DD.md`, creating the file
  > if it does not exist."
- **Option B — Pause sync during sessions:** Pause iCloud or Obsidian Sync
  during heavy ingest sessions, then let it sync when done.
- **Option C — Use Git instead of cloud sync:** Git merges are more reliable
  than iCloud conflict resolution for text files.

### ChatGPT forgetting the system prompt in long sessions

**Cause:** Very long conversations cause early system prompt content to fall
outside the model's effective attention window, even if it is technically in
context.

**Fix:**
1. Start a new conversation in the same Project (Project Instructions persist
   automatically).
2. Or paste the key section of `SYSTEM_PROMPT.md` as a new message:
   > "Here are your updated instructions: [paste relevant section]"
3. For Custom GPTs, this is less common — the system prompt is loaded
   afresh each conversation.

### Local models ignoring routing or formatting rules

**Cause:** The system prompt is too complex for the model size, or the model
has been fine-tuned for chat rather than instruction following.

**Fix:**
1. Use a larger model. Routing and structured output require at least 32B
   parameters for reliable results.
2. Generate a shorter system prompt: run the wizard and select compact output.
3. Simplify the routing table — reduce to 3–4 agents maximum. Merge domains
   if needed.
4. Add explicit instruction reinforcement after the system prompt:
   ```
   Remember: always write frontmatter in YAML format. Always save research
   output to raw/research/. Never skip the Karpathy enrichment pass.
   ```

### Frontmatter not rendering in Obsidian

**Cause:** Obsidian's Properties view may be disabled, or the frontmatter
is malformed (missing closing `---`, incorrect YAML indentation).

**Fix:**
1. In Obsidian, go to **Settings** → **Editor** → enable **Properties in
   document** (or **Show frontmatter**).
2. Check the generated page for frontmatter errors:
   - Both `---` delimiters must be present (opening and closing)
   - YAML keys must not have tabs — use spaces
   - List values must use `- item` format
   - Strings with colons must be quoted: `title: "My Title: Subtitle"`
3. If the LLM generates malformed frontmatter repeatedly, add an example
   to your system prompt with a note: "Always validate YAML syntax."

### Orphan pages reported by Lint

**Cause:** A concept or entity page was created during Ingest but no other
page links to it, or the source page's `## Concepts Extracted` section
is missing wikilinks.

**Fix:**
1. Run `Lint` to get a list of orphans.
2. For each orphan, determine where it should be linked:
   - Add a `[[wikilink]]` to the source page that created it
   - Add an entry to `wiki/index.md`
   - Add it to a related concept's `## Connections` section
3. Prevent future orphans: during Ingest, always complete Phase 3 of the
   ingest checklist (Karpathy enrichment pass) — this ensures every new page
   is cross-linked into at least one existing page.

---

### Research output not saved to raw/research/

**Cause:** The agent completed the research in context but did not write the
file, or wrote it to the wrong path.

**Fix:**
- For Claude Code: explicitly instruct after Research:
  ```
  Save the output to raw/research/YYYY-MM-DD-[topic].md
  ```
- For other platforms: the file cannot be written automatically. Ask the LLM
  to output the full content as a formatted code block, then create the file
  manually in Obsidian.
- Add to your system prompt: "Never complete a Research task without saving
  the output file. Always confirm the file path after saving."
